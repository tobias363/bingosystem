## 0. Open questions for reviewer (stopp her hvis svar trengs før kode)

### Q1 — Tabell-design: dedikert eller delt med OK Bingo (B3.5)?

Legacy bruker én collection `slotmachine` med `machineName: 'Metronia' | 'OK Bingo'`-discriminator. B3.5-plan bygger samme integrasjon for OK Bingo.

- **Mitt forslag:** Delt tabell `app_machine_tickets` med `machine_name CHECK IN ('METRONIA', 'OK_BINGO')`. Felles kolonner (player_id, hall_id, ticket_number, balance, is_closed, closed_at). OK-Bingo-spesifikke felter (room_id) som NULL eller via `other_data` JSONB. B3.5 trenger ingen ny tabell, bare ny api-klient.
- **Alternativ:** Egen `app_metronia_tickets` + `app_okbingo_tickets`. Ren scope-grense, men dupliserer schema og service-mønster.
- **Konsekvens av delt:** B3.4-migrasjonen oppretter tabellen + indekser. B3.5 muterer kun via samme service.

### Q2 — Dispatcher-pattern eller Metronia-spesifikke routes?

Legacy bruker generiske `/agent/create-ticket` med `machineName`-felt i body som dispatches. Brukers scope foreslår per-machine routes (`/api/agent/metronia/...`).

- **Mitt forslag:** Per-machine routes (per brukers scope). Enklere RBAC, OpenAPI-docs, og test-stubbing per integrasjon. Internal service-laget kan dele generiske `MachineTicketService` om ønskelig.
- **Alternativ:** Generisk `/api/agent/machines/:machineName/...`. Færre endepunkter (×2), men dårligere docs og tighter coupling.

### Q3 — Action_type-ny enums vs gjenbruk av eksisterende

B3.2 har `app_agent_transactions.action_type IN ('CASH_IN', 'CASH_OUT', 'TICKET_SALE', 'TICKET_REGISTER', 'TICKET_CANCEL', 'FEE', 'OTHER')`. Metronia ops trenger å logges der for unified audit-trail.

- **Mitt forslag:** Utvid CHECK med `'MACHINE_CREATE'`, `'MACHINE_TOPUP'`, `'MACHINE_CLOSE'`, `'MACHINE_VOID'`. Klarere audit-rapporter. 1-linjes migrasjon.
- **Alternativ:** Bruk eksisterende `TICKET_REGISTER`/`TICKET_CANCEL`/`CASH_OUT` med `otherData.machineName`-discriminator. Færre constants, men søker via JSONB blir tregere.

### Q4 — Cancel/Void-vindu (samme som B3.2 cancel)?

Brukers scope inkluderer `POST /void` for "angre salg". Legacy har ikke explicit void — agenten må bruke `closeTicketOfMachine` med remaining-balance refunds.

- **Mitt forslag:** Legg til void = "kanseller ticket innen 5 min etter create" (kortere vindu enn B3.2 cancel siden Metronia-API allerede har debitert). For ticket > 5 min: bruk close-ticket (med restbeløp-refund). ADMIN_FORCE for void utover vinduet.
- **Alternativ:** Drop void-endepunktet; alle "angre" kjører gjennom close-ticket. Enklere semantikk, men dårligere UX for "agenten skrev feil amount → angre med en gang".
- **Konsekvens av void:** En ekstra endepunkt + 2-3 tester. Reuser counter-tx-mønster fra B3.2.

### Q5 — HTTP-klient: fetch + cert-håndtering

Legacy bruker `axios` med `rejectUnauthorized: false` (per-request cert-bypass). Sikkerhets-issue.

- **Mitt forslag:** Bruk Node `fetch`. Default: rejectUnauthorized=true (production). Hvis Metronia bruker self-signed cert i staging, eksplisitt `METRONIA_TLS_REJECT_UNAUTHORIZED=false` env-var (default true). Aldri silent bypass i prod.
- **Alternativ:** Behold axios for 1:1 paritet. Men axios er dependency vi unngår om mulig (per repo-konvensjon).

### Q6 — Stub-strategi i tester

- **Mitt forslag:** `MetroniaApiClient`-interface + `StubMetroniaApiClient` for tester. Real `HttpMetroniaApiClient` brukes i prod. Stub returnerer predictable success/error per ticket-number. Ingen ekte API-kall i CI.
- **Alternativ:** Mock fetch globalt med MSW eller nock. Mer realistisk men ekstra dep.

## 3. Nye tabeller — schema + migrasjoner

### 3.1 `20260419100000_machine_tickets.sql`

```sql
-- BIN-583 B3.4: machine-tickets — port av legacy slotmachine.
--
-- Felles tabell for Metronia (B3.4) og OK Bingo (B3.5). machine_name
-- diskriminerer. OK-Bingo-spesifikt felt room_id holdes som kolonne;
-- andre machine-spesifikke felter via other_data JSONB.

CREATE TABLE IF NOT EXISTS app_machine_tickets (
  id                    TEXT PRIMARY KEY,
  machine_name          TEXT NOT NULL CHECK (machine_name IN ('METRONIA', 'OK_BINGO')),
  ticket_number         TEXT NOT NULL,
  external_ticket_id    TEXT NOT NULL,        -- API-returnert ticket-ID (Metronia ticketId)
  hall_id               TEXT NOT NULL REFERENCES app_halls(id) ON DELETE RESTRICT,
  shift_id              TEXT NULL REFERENCES app_agent_shifts(id) ON DELETE SET NULL,
  agent_user_id         TEXT NOT NULL REFERENCES app_users(id) ON DELETE RESTRICT,
  player_user_id        TEXT NOT NULL REFERENCES app_users(id) ON DELETE RESTRICT,
  room_id               TEXT NULL,            -- For OK Bingo (B3.5); null for Metronia
  initial_amount_cents  BIGINT NOT NULL CHECK (initial_amount_cents >= 0),
  total_topup_cents     BIGINT NOT NULL DEFAULT 0,
  current_balance_cents BIGINT NOT NULL DEFAULT 0,
  payout_cents          BIGINT NULL,          -- Settet ved close
  is_closed             BOOLEAN NOT NULL DEFAULT false,
  closed_at             TIMESTAMPTZ NULL,
  closed_by_user_id     TEXT NULL REFERENCES app_users(id) ON DELETE SET NULL,
  void_at               TIMESTAMPTZ NULL,
  void_by_user_id       TEXT NULL REFERENCES app_users(id) ON DELETE SET NULL,
  void_reason           TEXT NULL,
  unique_transaction    TEXT NOT NULL UNIQUE, -- Idempotency-nøkkel sendt til Metronia API
  other_data            JSONB NOT NULL DEFAULT '{}'::jsonb,
  created_at            TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at            TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (machine_name, ticket_number)
);

CREATE INDEX IF NOT EXISTS idx_app_machine_tickets_hall_machine
  ON app_machine_tickets(hall_id, machine_name, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_app_machine_tickets_player
  ON app_machine_tickets(player_user_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_app_machine_tickets_shift
  ON app_machine_tickets(shift_id, created_at DESC)
  WHERE shift_id IS NOT NULL;
CREATE INDEX IF NOT EXISTS idx_app_machine_tickets_open
  ON app_machine_tickets(machine_name, hall_id)
  WHERE is_closed = false;

COMMENT ON TABLE app_machine_tickets IS
  'BIN-583 B3.4/B3.5: ticket-rader for eksterne maskin-integrasjoner (Metronia, OK Bingo).';
COMMENT ON COLUMN app_machine_tickets.unique_transaction IS
  'Idempotency-nøkkel sendt til ekstern API (Metronia transaction-felt). UNIQUE forhindrer dobbel-create.';
COMMENT ON COLUMN app_machine_tickets.payout_cents IS
  'Beløp utbetalt ved close (current_balance ved close-tidspunktet). NULL for åpne tickets.';
```

### 4.1 `MetroniaApiClient.ts`

```ts
/**
 * BIN-583 B3.4: Metronia API-klient.
 *
 * HTTP-kall mot Metronia-systemet for ticket-create/topup/close/status.
 * Real-impl bruker fetch; tester injecterer StubMetroniaApiClient.
 *
 * Env-vars (kun lest av wirefil index.ts; klienten tar verdier som constructor-arg):
 *   METRONIA_API_URL               base-URL (f.eks. https://metronia.example.com/api/v1)
 *   METRONIA_API_TOKEN             Bearer token
 *   METRONIA_TLS_REJECT_UNAUTHORIZED  default 'true'; sett 'false' for self-signed staging
 *   METRONIA_TIMEOUT_MS            default 10_000
 */

export interface MetroniaCreateTicketInput {
  amountCents: number;          // I cents (Metronia regner i øre)
  uniqueTransaction: string;    // Idempotency-nøkkel
}
export interface MetroniaCreateTicketResult {
  ticketNumber: string;
  ticketId: string;
}

export interface MetroniaTopupInput {
  ticketNumber: string;
  amountCents: number;
  uniqueTransaction: string;
}
export interface MetroniaCloseInput {
  ticketNumber: string;
  uniqueTransaction: string;
}
export interface MetroniaCloseResult {
  finalBalanceCents: number;    // Hva som skal credited tilbake til player
}
export interface MetroniaStatusResult {
  balanceCents: number;
  ticketEnabled: boolean;
  isReserved: boolean;
}

export interface MetroniaApiClient {
  createTicket(input: MetroniaCreateTicketInput): Promise<MetroniaCreateTicketResult>;
  topupTicket(input: MetroniaTopupInput): Promise<{ newBalanceCents: number }>;
  closeTicket(input: MetroniaCloseInput): Promise<MetroniaCloseResult>;
  getStatus(ticketNumber: string): Promise<MetroniaStatusResult>;
}

// HttpMetroniaApiClient: real-impl via fetch
// StubMetroniaApiClient: in-memory for tester (predictable success/error per ticket-number)
```

### 4.3 Service-orkestrering

`MetroniaTicketService.ts` (i `src/agent/`):
- `createTicket({ agentUserId, playerUserId, amountCents })` — debit wallet, call API, insert DB, log tx
- `topupTicket({ agentUserId, ticketId, amountCents })` — debit wallet, call API, update DB, log tx
- `closeTicket({ agentUserId, ticketId })` — call API, get final balance, credit wallet, update DB, log tx
- `voidTicket({ agentUserId, ticketId, reason })` — within 5 min: refund + close, mark voided
- `getDailySalesForShift(agentUserId)` — aggregate Metronia-tx for current shift
- `getHallSummary(hallId, fromDate, toDate)` — aggregate per hall (admin)
- `getDailyReport(date)` — aggregate global (admin)

Reuses `AgentTransactionStore.insert()` for unified audit, `WalletAdapter` for player-wallet, `AgentSettlementService.shift-frozen-check` (via shift-state).
