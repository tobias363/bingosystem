## 0. Open questions for reviewer (KRITISK Q1 før kode)

### Q1 — OK Bingo legacy = SQL Server-polling, ikke HTTP

**FUNN:** Legacy `createOkBingoAPI` (`machineApiController.js:599`) er IKKE et HTTP REST-API som Metronia. Det er en **SQL Server-tabell-basert RPC**:
1. INSERT rad i `COM3`-tabell med `BingoID`, `ComandID`, `Parameter` (semicolon-separert string)
2. Poll samme tabell hver 1 sek (max 10 forsøk) for response-rad (matchende `ComID` + `ComandID + 100`)
3. Parse semicolon-string fra `Parameter`-felt
4. Krever `mssql`-pakke + SQL Server connection-string

Dette er fundamentalt annerledes fra Metronia (REST + Bearer token). User-scope sa "gjenbruk MachineApiClient-interface" men Metronia-interface passer ikke for SQL-polling-protokollen.

**Tre forslag — velg én:**

- **(a) Stub-only "real" impl + flag som integration-pending** *(mitt forslag)*
  - Bygg `OkBingoApiClient` interface (matcher OK Bingo's domene-konsept: roomId + ticketNumber)
  - `StubOkBingoApiClient` for tester + lokal-dev (mirror Metronia-stub)
  - `RealOkBingoApiClient` kaster `OKBINGO_NOT_CONFIGURED` ved alle metoder — eksisterer kun for at index.ts-wiring kompilerer
  - Endepunktene fungerer mot Stub i CI; produksjon trenger separat decision (SQL Server bridge ELLER OK Bingo går HTTP)
  - Plan: BIN-XXX follow-up for real integrasjon når OK Bingo-team gir HTTP-API ELLER vi godkjenner SQL Server bridge
  - **Pro:** raskt levert, alle endepunkter funksjonelle, gates grønne. **Con:** ikke prod-klar uten follow-up.

- **(b) Full SQL Server polling-impl med `mssql`-dep**
  - Port hele legacy-protokollen 1:1 (INSERT + poll + parse)
  - Krever ny dep `mssql` + `OKBINGO_DB_CONNECTION_STRING` env
  - **Pro:** prod-klar fra dag én. **Con:** 2-3 dager ekstra arbeid + nye deps + ny CI-test-dependency

- **(c) Vent på OK Bingo HTTP-API-decision**
  - Blokker B3.5 til strategisk avgjørelse er tatt
  - **Pro:** ingen wasted work. **Con:** blokkerer leveranse-kadens; B3.6/B3.7 + B3.8 går videre uten

### Q2 — open-day-endepunktet (commandId=11)

OK Bingo har en spesiell "open day"-signal som agent sender ved start-of-day for å initialize hardware-runden. Metronia har ingen tilsvarende.

- **Mitt forslag:** Egen endepunkt `POST /api/agent/okbingo/open-day` som kaller `OkBingoApiClient.openDay()`. Logges som audit-event (ingen tx, ingen wallet). 9 endepunkter totalt.
- **Alternativ:** Auto-kall ved første ticket-create i shift. Mindre eksplisitt; UX-forenkling, men avviker fra legacy-pattern.

### Q3 — `roomId`-validering

Hver OK Bingo-ticket er bundet til en `roomId` (bingo-room-hardware-ID, default 247 i legacy). Per hall er det vanligvis én room.

- **Mitt forslag:** Lagre `roomId` på `app_machine_tickets.room_id` (kolonnen finnes fra B3.4). Default 247 hvis ikke spesifisert; configurable per hall via `app_halls.other_data.okbingoRoomId` (port til egen migration hvis ønskelig). For B3.5 hardcoder vi 247 som default + tar `roomId` i request body som override.
- **Alternativ:** Kreve `roomId` i hver request — eksplisitt, ingen magic. Mer boilerplate for agent.

### Q4 — Tabell- og service-arkitektur — gjenbruke MetroniaTicketService?

`MetroniaTicketService` har det samme orkestrerings-mønstret (wallet-debit → API → DB+tx). Kunne refaktoreres til generisk `MachineTicketService<TClient>`.

- **Mitt forslag:** Ny `OkBingoTicketService` som speiler MetroniaTicketService 1:1 med OkBingo-spesifikke felter (roomId, commandId-mapping). Refactor til generisk service kan tas som BIN-XXX follow-up når begge har stabilisert. Ingen forhastet abstraksjon.
- **Alternativ:** Refactor til generisk service nå. Risiko: feilabstraksjon hvis B3.5 viser uventede forskjeller.

### 4.1 `OkBingoApiClient.ts` (interface)

```ts
/**
 * BIN-583 B3.5: OK Bingo API-klient — interface + factory.
 *
 * NB: Real-impl er BLOKKET av Q1 (legacy bruker SQL Server polling, ikke
 * REST). For B3.5 leveres Stub som default; "real" eksisterer som
 * NOT_IMPLEMENTED-throw inntil arkitektur-decision.
 */

export interface OkBingoCreateTicketInput {
  amountCents: number;
  roomId: number;
  uniqueTransaction: string;
}
export interface OkBingoCreateTicketResult {
  ticketNumber: string;
  ticketId: string;
  roomId: number;
}

export interface OkBingoTopupInput {
  ticketNumber: string;
  amountCents: number;
  roomId: number;
  uniqueTransaction: string;
}
export interface OkBingoCloseInput {
  ticketNumber: string;
  roomId: number;
  uniqueTransaction: string;
}
export interface OkBingoStatusResult {
  balanceCents: number;
  ticketEnabled: boolean;
}

export interface OkBingoApiClient {
  createTicket(input: OkBingoCreateTicketInput): Promise<OkBingoCreateTicketResult>;
  topupTicket(input: OkBingoTopupInput): Promise<{ newBalanceCents: number }>;
  closeTicket(input: OkBingoCloseInput): Promise<{ finalBalanceCents: number }>;
  getStatus(ticketNumber: string, roomId: number): Promise<OkBingoStatusResult>;
  openDay(roomId: number): Promise<{ opened: true }>;
}
```
