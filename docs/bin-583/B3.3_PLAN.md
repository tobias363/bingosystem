## 0. Open questions for reviewer (stopp her hvis svar trengs før kode)

### Q1 — Settlement-state-modell: egen tabell eller JSONB på shift?

Legacy bruker dedikert `settlement`-collection + `agentShift.settlement` JSONB + `hall.otherData.isPreviousDaySettlementPending`. Tre steder som må holdes i sync.

- **Mitt forslag:** Dedikert `app_agent_settlements`-tabell med core cash-count-felter (opening/closing balance, diff, drop-safe, bilags-note). Machine-specific revenue (Metronia, OK Bingo, Norsk Tipping, etc.) parkeres til B3.4/B3.5 som JSONB-extension via `other_data`. `app_agent_shifts.settled_at` timestamp settes atomisk for freeze-enforcement.
- **Alternativ:** Bruk `shifts.settlement JSONB` direkte (eksisterer fra B3.1). Enklere, men ingen relational integrity og tregere aggregat-rapporter (B3.8 vil trenge disse).
- **Konsekvens av dedikert tabell:** ~80-linjers migrasjon + FK til `app_agent_shifts`. Fremtidige B3.4/B3.5 utvider via JSONB-kolonne, ikke ALTER.

### Q2 — Transactions-freeze: hvor sjekkes det?

- **Mitt forslag:** Sjekk i `AgentTransactionService` (cashIn/cashOut/sellPhysical/cancel) — `if (shift.settled_at) throw SHIFT_SETTLED`. Defense-in-depth: ingen partial unique-index på DB-nivå (shift.id + transaction-insertion). Bare service-layer-enforcement + auditlog av forsøk.
- **Alternativ:** DB-trigger som blokkerer INSERT på `app_agent_transactions` hvis relatert shift har `settled_at IS NOT NULL`. Robustere, men Postgres-trigger-vedlikehold er en byrde.
- **Konsekvens av service-layer:** Krever at alle mutation-paths går gjennom AgentTransactionService. Allerede tilfelle fra B3.2.

### Q3 — Shift/end vs shift/close-day: lifecycle-ordning

I B3.1 har vi `shift/end` som setter `is_active=false`. Legacy lar agenten logge ut UTEN å settle — settlement kan gjøres senere (evt. av neste-dags agent).

- **Mitt forslag:** Behold begge som distinkte steg:
  1. `POST /api/agent/shift/end` (eksisterer fra B3.1) — avslutter shift, men settlement kan gjøres senere
  2. `POST /api/agent/shift/close-day` (nytt) — krever at shift EITHER er aktiv OR ended uten settlement. Setter `settled_at`, skriver settlement-rad, transfererer daily_balance til hall_cash_balance.
- **Alternativ A:** Slå sammen. `shift/end` tar valgfrie cash-count-felter; hvis med → full settlement. Enklere API, men blander to konsepter.
- **Alternativ B:** Close-day er auto-forced ved neste-dags `shift/start` hvis forrige shift ikke er settled. Legacy gjør dette med `isPreviousDaySettlementPending`-flag. Port vs. fail-fast.
- **Konsekvens:** Alternativ (min forslag) matcher legacy-kontrakten uten å tvinge API-forenkling. Reviewer kan velge annerledes.

### Q4 — Control-daily-balance: pre-close eller integrert?

Legacy har `controlDailyBalance` som egen handler — agent oppgir påstått dailyBalance + totalCashBalance før settlement. Diff lagres på `shift.controlDailyBalance` JSONB.

- **Mitt forslag:** Eget endepunkt `POST /api/agent/shift/control-daily-balance` som skriver til `shifts.control_daily_balance JSONB` og lager audit-log. Ingen wallet-mutasjon. Kan kalles flere ganger (overskriver).
- **Alternativ:** Inkluder som optional pre-step i `close-day`-body. Færre endepunkter men mindre fleksibelt (agent kan ikke ta "kaffepause-pluss-sjekk" uten å commite).
- **Konsekvens av separat:** 1 ekstra endepunkt, men matcher legacy-UX.

### Q5 — Cash-diff: feil-behandling

Når agentens rapporterte kontantbeholdning avviker fra shift.daily_balance, hva skal skje?

- **Mitt forslag:** Tillat close-day med diff (både positiv og negativ). Diff lagres i `settlement.daily_balance_difference`. Hvis diff > konfigurerbar terskel (default 500 kr) eller > 5 %, krever note-felt. Audit-logges med `agent.settlement.diff-large` action. Admin får rapporter via B3.8.
- **Alternativ A:** Block close-day hvis diff > terskel; kreve admin-approve først. Strengere, men blokkerer drift. Legacy ikke så streng.
- **Alternativ B:** Ingen terskel, ingen note-krav. Dårlig for compliance.

### Q6 — Edit-settlement post-close

Legacy har `editSettlement` som lar admin korrigere en settled shift.

- **Mitt forslag:** Port som `PUT /api/admin/shifts/:shiftId/settlement` — kun ADMIN (AGENT_SHIFT_FORCE permission). Lager counter-tx på eventuelle wallet-differences + auditlog med full før/etter-diff. Krever note-felt.
- **Alternativ:** Drop edit — kreve reverse-settlement + ny close-day hvis feil. Renere konseptuelt, men bryter 100% paritet.
- **Konsekvens:** Separat endepunkt, estimert +40 linjer + 2 tester.

## 6. Transactions-freeze-mekanisme

Endring i `AgentTransactionService`:

```ts
// I alle mutation-paths (cashIn, cashOut, sellPhysical, cancelPhysicalSale):
private async requireActiveShift(agentUserId: string) {
  await this.agents.requireActiveAgent(agentUserId);
  const shift = await this.shifts.getCurrentShift(agentUserId);
  if (!shift) throw new DomainError("NO_ACTIVE_SHIFT", "...");
  if (shift.settledAt) {
    throw new DomainError(
      "SHIFT_SETTLED",
      "Denne shiften er avsluttet med settlement. Åpne ny shift for nye transaksjoner."
    );
  }
  return shift;
}
```

Defense-in-depth: vi stoler på service-layer siden alle write-paths går gjennom dette. Ingen DB-trigger.

**Cancel-edge-case:** Cancel av ticket-sale utenfor 10-min-vindu krever ADMIN_FORCE (fra B3.2). Etter settlement er vinduet implisitt stengt — cancel feiler med `SHIFT_SETTLED` før den sjekker vindu.

## 8. Cash-diff: threshold-logikk

Konstant i `AgentSettlementService`:

```ts
export const DIFF_WARN_THRESHOLD_NOK = 500;
export const DIFF_WARN_THRESHOLD_PCT = 5;  // 5 % av daily_balance_at_end
```

- Diff innenfor terskel → close-day OK uten note
- Diff over terskel → note-felt required, ellers 400 `DIFF_NOTE_REQUIRED`
- Diff over 2× terskel → `forced=true` + ADMIN_FORCE-permission required
