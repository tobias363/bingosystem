## 1. BIR-036 — Kontant-utbetaling-cap per hall

### Eier-avklaring

> "Det er ingen begrensninger på hvor mye som kan utbetales av premier
> gjennom systemet hvor det da overføres til bankkonto. Men 50 000 kr maks
> i kontanter utbetaling [per hall, per dag]."

### Konkret krav

| Utbetaling-type | Daglig grense per hall |
|---|---|
| **Bankoverføring** (`Withdraw in Bank` → XML-eksport) | **Ingen grense** |
| **Kontant** (`Withdraw in Hall` → agent gir kontanter ut) | **50 000 kr** |

### Hva som finnes i dag

- `WithdrawXmlExportService.ts` (`apps/backend/src/admin/`) — bank-pipeline (XML-eksport til regnskap-mail). **Ingen cap-sjekk.**
- `Withdraw in Hall` accept-flyt i `paymentRequests`-routes — **ingen cap-sjekk.**
- `app_halls.cash_balance` (HallCashLedger) — eksisterer, men brukes til agent-kasse-settlement, ikke til withdraw-cap.

### Hva som mangler

1. **Per-hall daglig kontant-counter:** ny tabell eller agreggert query
   - Anbefaling: ny tabell `app_hall_cash_withdrawals_daily` (hall_id, business_date, total_amount_cents, count) — atomisk increment per accept
   - Alternativ: aggreger fra `app_payment_requests` der `kind=withdraw` AND `accepted` AND `destination_type=hall_cash` på business_date.
2. **Cap-sjekk i HALL_OPERATOR/AGENT accept-stien:**
   - `POST /api/admin/payments/requests/:id/accept` (admin) og agent-versjon
   - Før wallet-debit: hent dagens akkumulerte kontant-utbetaling for hallen, sjekk `accumulated + new_amount ≤ 50_000`
   - Hvis over → fail-closed med `CASH_WITHDRAW_CAP_EXCEEDED` + remaining-budget i feilmeldingen
3. **Bank-utbetaling forblir uberørt** (separat kode-sti).
4. **Audit-log entry:** `cash_withdrawal.cap_exceeded` for revisjon.
5. **Admin-override (post-pilot):** ADMIN-rolle kan overstyre cap med påkrevd `reason` (skrives til audit).

### Estimat

**~1-2 dev-dager** (DB-tabell/migration + cap-sjekk i 2 endepunkter + 5-8 unit-tester).

### Sized PR

- **PR HV2-A:** DB-migrasjon + `HallCashWithdrawalCapService` + cap-sjekk i admin/agent accept-handler + audit-log + tests.

## 2. HV-2 — Spill 1 default-gevinst-floors

### Eier-avklaring

> "Det vil alltid bli satt en default gevinst for rad 1, 2, 3, 4 og fullt
> hus. Uavhengig av hvor mange som spiller så gjelder disse default-
> verdiene. Avhengig av hvilket av spillene som spilles så kan da disse
> default-verdiene endres ut fra hva som er satt inn som innstilling på
> det spillet, men det kan da aldri gå under disse default-verdiene. Man
> må da ha mulighet til å sette disse default-verdiene. Slik er det vel i
> legacy backend i dag også når det opprettes spill."

> "På Spill 2 og 3 er det variabelt utfra hvor mye som er kjøpt av bonger.
> Så vi trenger ikke å skrote den funksjonaliteten — må bare sette den
> til side og implementere på Spill 2 og 3."

### Tolket krav (Spill 1 — slug `bingo`)

| Pattern | Default-floor | Per-spill-override |
|---|---|---|
| Rad 1 (Phase 1) | konfigurerbar | ≥ default |
| Rad 2 (Phase 2) | konfigurerbar | ≥ default |
| Rad 3 (Phase 3) | konfigurerbar | ≥ default |
| Rad 4 (Phase 4) | konfigurerbar | ≥ default |
| Fullt Hus (Phase 5) | konfigurerbar | ≥ default |

**Bindende regel:**
- `actual_payout = max(default_floor, computed_from_pool)`
- Default-floor utbetales **uavhengig** av antall spillere / pool-størrelse — hallen pre-funds gapet hvis pool < floor.
- Override per spill (Wheel of Fortune, Treasure Chest, Mystery, ColorDraft, etc.) kan ØKE floor men aldri senke den under hall-default.

### Spill 2/3 (slug `rocket`, `monsterbingo`)

- **Beholder** dagens variable-by-ticket-count payout-modell (Game2JackpotTable, etc.).
- **Ingen** floor-mekanisme. Dette er et bevisst skille — ikke implementeres på Spill 2/3.

### Hva som finnes i dag

| Komponent | Status |
|---|---|
| Per-pattern `minPrize`-felt på `PatternDefinition` | ✅ `apps/backend/src/game/variantConfig.ts` |
| Engine bruker `Math.max(rawPhase, phase.minPrize)` | ✅ `apps/backend/src/game/BingoEnginePatternEval.ts:291,360` |
| Sub-variant defaults (`SPILL1_SUB_VARIANT_DEFAULTS`) | ✅ `packages/shared-types/src/spill1-sub-variants.ts` |
| `spill1Overrides` propagering (per-spill custom floor) | ✅ `spill1VariantMapper.ts:223` |
| **Demo Hall RTP-cap-bypass** så default alltid utbetales | ✅ `PhasePayoutService.ts:216` (`isTestHallRtpBypass`) |
| **Prod hall: house pre-funds gap når pool < floor** | ❌ MANGLER — RTP-cap kan fortsatt redusere payout under floor |
| **Admin-UI for å sette/redigere defaults** | 🟡 Delvis (sub-variant-presets eksisterer; per-pattern-floor-edit må verifiseres) |

### Hva som mangler

1. **Prod-hall house-pre-fund:** `PhasePayoutService.ts:222-225` capper på `remainingPayoutBudget` (RTP-budget). Når RTP < minPrize, blir payout < minPrize for prod-haller. Eier-krav: floor MÅ alltid utbetales.
   - **Fix-strategi:** når `requestedAfterPolicyAndPool < minPrize` AND `houseAvailableBalance ≥ minPrize`, bruk `minPrize` som faktisk payout og finansier gapet fra `houseAccountId` (hallens cash-balance). Dette ER HV-2 Option A.
   - **Compliance-implikasjon:** ledger-entry må vise STAKE-pool og PRIZE separat — STAKE kan være 100kr, PRIZE 200kr, hall finansierer 100kr-gap. Compliance ledger skal allerede støtte dette via `house_account` source.
2. **Per-hall default-tabell:** ny `app_spill1_prize_defaults` (hall_id, pattern_phase, minPrize_cents, updated_at, updated_by) — eller utvid `app_game_settings`.
3. **Admin-UI** for å sette/redigere per-hall defaults med constraint-validation (per-spill-override må være ≥ default).
4. **Override-håndhevelse i `Game1ScheduleService`:** når et spesifikt spill (sub-variant) er valgt, sjekk at `overridePhaseN.minPrize ≥ defaultPhaseN.minPrize` — fail-closed på overstyring under default.
5. **Demo Hall fortsetter å bruke bypass** — ingen endring der.

### Estimat

**~3-4 dev-dager:**
- DB-migrasjon for per-hall defaults: 0.5d
- Backend service + cap-håndhevelse i `PhasePayoutService`: 1d
- Admin-UI (CRUD + validation): 1d
- Tests: 0.5-1d
- Migration av eksisterende `SPILL1_SUB_VARIANT_DEFAULTS` til per-hall-defaults: 0.5d

### Sized PR-er

- **PR HV2-B1:** DB-migrasjon + `Spill1PrizeDefaultsService` + read-API
- **PR HV2-B2:** PhasePayoutService — house-pre-fund gap-logikk + tests
- **PR HV2-B3:** Admin-UI for editing per-hall defaults + override-validation
- **PR HV2-B4:** ScheduleService — håndhev override ≥ default ved spill-opprettelse
